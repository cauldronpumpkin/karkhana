# Implementation Plan: Cost Attribution and Token Tracking System

This plan adds project-, phase-, worker-, model-, and provider-level cost visibility to Karkhana.

Target stack:

- Backend: FastAPI, Python 3.11+
- Frontend: Svelte 5
- Storage: DynamoDB single-table
- Execution: local worker bridge
- Providers: OpenAI, Anthropic, others through direct clients and/or `codex-lb` LiteLLM proxy
- Existing proxy: `codex-lb` already tracks token usage at proxy level

---

# 1. Data Model

## 1.1 Core Concepts

The system should track every billable LLM usage as a normalized `CostEvent`.

Each event should answer:

- Which project?
- Which phase?
- Which worker or user initiated it?
- Which provider/model?
- How many input/output/cache tokens?
- What did it cost?
- Was it estimated or confirmed?
- What request generated it?
- Was it routed through LiteLLM, direct provider SDK, or local worker?

---

```python
  project_id = if_not_exists(project_id, :project_id),
  updated_at = :updated_at
"""
```

Use integer micros for aggregation:

```text
total_cost_micros = int(total_cost_usd * 1_000_000)
```

This avoids Decimal update issues and rounding drift.

Display conversion:

```python
usd = Decimal(total_cost_micros) / Decimal(1_000_000)
```

# Key Design Decisions

1. **Use raw events for audit, aggregates for dashboard speed.**
2. **Use idempotency keys to prevent double counting across LiteLLM and workers.**
3. **Use integer cost micros in aggregates to avoid Decimal drift.**
4. **Prefer LiteLLM-provided cost when available, otherwise calculate from local pricing.**
5. **Enforce budgets before LLM execution, update actual spend after completion.**
6. **Make phase, worker, model, and provider first-class attribution dimensions.**
7. **Keep quality optional initially, but include fields now so model comparison can evolve naturally.**
