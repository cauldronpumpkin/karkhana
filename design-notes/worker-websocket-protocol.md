# Worker WebSocket Streaming Protocol Implementation Plan for Karkhana

This plan adds a WebSocket-based real-time streaming protocol to Karkhana for:

1. Worker job status streaming: `claimed → running → progress → complete/fail`
2. Frontend live dashboard updates
3. Worker and dashboard authentication
4. Reconnection, heartbeat, and message validation
5. FastAPI backend integration
6. Svelte 5 frontend integration
7. Rust/Tauri local worker integration

Assumed stack:

- Backend: FastAPI, Python 3.11+
- Frontend: Svelte 5
- Database: DynamoDB single-table
- Worker: Tauri v2 + Rust desktop application
- Existing REST APIs for worker registration, heartbeat, claim, completion/failure
- Workers authenticate using hashed tokens stored server-side

# 1. Message Protocol

## 1.1 Protocol Overview

All WebSocket messages are JSON objects with a common envelope.

Every message must include:

```json
{
  "type": "message.type",
  "version": 1,
  "message_id": "uuid-or-client-generated-id",
  "timestamp": "2026-05-05T12:00:00.000Z",
  "payload": {}
}
```

Recommended semantics:

- `type`: String identifier for routing.
- `version`: Protocol version. Start with `1`.
- `message_id`: Unique per message.
- `timestamp`: ISO-8601 UTC timestamp generated by sender.
- `payload`: Type-specific object.

Backend should validate every inbound message.

Frontend and worker clients should ignore unknown message types but log them.

---

## 1.2 Connection Roles

There are two WebSocket client roles:

```ts
type WebSocketRole = "worker" | "dashboard";
```

### Worker connection

Used by local worker apps to:

- Authenticate as a registered worker
- Send lifecycle events
- Send job status updates
- Receive job assignment/control messages in future
- Receive ping/pong heartbeat messages

### Dashboard connection

Used by Svelte frontend to:

- Subscribe to live worker/job activity
- Receive worker status updates
- Receive job status/progress/completion events
- Receive aggregate dashboard snapshots

## 1.3 WebSocket Endpoints

Use two endpoints for clarity:

```txt
/ws/workers
/ws/dashboard
```

Alternative: one endpoint `/ws` with `role` in auth payload. But two endpoints simplify routing and authorization.

Recommended implementation:

```txt
backend/app/api/ws_workers.py
backend/app/api/ws_dashboard.py
```

Register both routers in the FastAPI app.

## 1.4 Common Envelope Schema

Create this file:

```txt

```

Pydantic models:

```python
from datetime import datetime
from typing import Any, Literal
from pydantic import BaseModel, Field


class WsMessage(BaseModel):
    type: str
    version: Literal[1] = 1
    message_id: str = Field(min_length=1, max_length=128)

    payload: dict[str, Any]
```

## 1.5.1 `server.hello`

Sent immediately after WebSocket accept.

### Direction

```txt
server → worker
server → dashboard
```

### Purpose

Confirms socket accepted and tells client protocol expectations.

### Payload

```json
{
  "connection_id": "conn_abc123",
  "server_time": "2026-05-05T12:00:00.000Z",
  "protocol_version": 1,
  "heartbeat_interval_ms": 30000,
  "auth_timeout_ms": 10000
}
```

### Full Message

```json
{
  "type": "server.hello",
  "version": 1,
  "message_id": "srv-msg-001",
  "timestamp": "2026-05-05T12:00:00.000Z",
  "payload": {
    "connection_id": "conn_abc123",
    "server_time": "2026-05-05T12:00:00.000Z",
    "protocol_version": 1,
    "heartbeat_interval_ms": 30000,
    "auth_timeout_ms": 10000
  }
}
```

### JSON Schema

```json
{
  "$id": "ServerHelloPayload",
  "type": "object",
  "required": [
    "connection_id",
    "server_time",
    "protocol_version",
    "heartbeat_interval_ms",
    "auth_timeout_ms"
  ],
  "properties": {
    "connection_id": { "type": "string" },
    "server_time": { "type": "string", "format": "date-time" },
    "protocol_version": { "type": "integer", "const": 1 },
    "heartbeat_interval_ms": { "type": "integer", "minimum": 5000 },
    "auth_timeout_ms": { "type": "integer", "minimum": 1000 }
  },
  "additionalProperties": false
}
```

# 2. Auth Flow

## 2.1 Worker WebSocket Auth Flow

Endpoint:

```txt
GET /ws/workers
```

Sequence:

1. Worker opens WebSocket connection.
2. Server accepts connection.
3. Server sends `server.hello`.
4. Worker must send `auth.worker` within `auth_timeout_ms`, recommended `10000`.
5. Server verifies:
   - Worker exists in DynamoDB.
   - Worker token hash matches stored token hash.
   - Worker is not disabled/revoked.
6. On success:
   - Server marks worker as `online` or `idle`.
   - Server records `last_seen_at`.
   - Server binds `connection_id → worker_id`.
   - Server sends `auth.accepted`.
   - Server broadcasts `worker.status.updated` to dashboards.
7. On failure:
   - Server sends `auth.rejected`.
   - Server closes socket with appropriate close code.

Recommended close codes:

```txt
1008 Policy Violation: auth failed
1002 Protocol Error: malformed auth message
1011 Internal Error: server failure
```

## 2.4 Authorization Rules

Worker sockets:

- Can only send messages for their authenticated `worker_id`.
- Can only update jobs assigned to that worker.
- Cannot subscribe to dashboard topics.
- Cannot claim arbitrary ownership through WebSocket unless backend verifies claim.

Dashboard sockets:

- Can subscribe to job/worker/log topics.
- Must be authorized by organization/project scope if Karkhana has multi-tenancy.
- Should not send worker job mutation messages.

# 3. Connection Lifecycle

## 3.1 Initial Connection Lifecycle

Worker lifecycle:

```txt
CONNECT /ws/workers
server → server.hello
worker → auth.worker
server → auth.accepted
server/dashboard broadcast → worker.status.updated
steady state
```

Dashboard lifecycle:

```txt
CONNECT /ws/dashboard
server → server.hello
dashboard → auth.dashboard
server → auth.accepted
dashboard → dashboard.subscribe
server → dashboard.snapshot
steady state
```

---

## 3.2 Heartbeat

Use application-level heartbeat because browser WebSocket clients cannot reliably access protocol-level ping/pong.

Configuration:

```python
WS_HEARTBEAT_INTERVAL_SECONDS = 30
WS_PONG_TIMEOUT_SECONDS = 10
WS_AUTH_TIMEOUT_SECONDS = 10
```

Server behavior:

1. Every `30s`, send `server.ping` to authenticated connections.
2. Track `last_pong_at`.
3. If no `client.pong` within `10s`, close the socket.
4. For worker connections, mark worker as `offline` if connection is the active connection for that worker.

Client behavior:

1. On `server.ping`, immediately send `client.pong`.
2. If socket closes unexpectedly, reconnect with backoff.

---

## 3.3 Reconnection Strategy

Both worker and dashboard clients should reconnect automatically.

Backoff:

```txt
initial delay: 500ms
factor: 2
max delay: 30000ms
jitter: ±20%
reset after stable connection for 60s
```

Pseudo:

```ts
delay = min(maxDelay, initialDelay * 2 ** attempts)
delay = delay * randomBetween(0.8, 1.2)
```

---

## 3.4 Worker Reconnection Semantics

When a worker reconnects:

1. It authenticates again using `auth.worker`.
2. Server checks if another active connection already exists for `worker_id`.
3. Recommended behavior:
   - Replace older connection with newer connection.
   - Close older socket with code `4001` and reason `Replaced by newer worker connection`.
4. Server loads worker current state from DynamoDB.
5. If worker had `current_job_id`, server should not automatically fail the job.
6. Worker should resume sending status for any running job.
7. Backend should mark missed updates based on eventual REST completion/failure if needed.

Custom close codes:

```txt
4001 REPLACED_CONNECTION
4002 AUTH_TIMEOUT
4003 VALIDATION_ERROR
4004 RATE_LIMITED
4005 WORKER_JOB_OWNERSHIP_ERROR
```

---

## 3.5 Dashboard Reconnection Semantics

Dashboard reconnection:

1. Re-authenticate.
2. Re-send `dashboard.subscribe`.
3. Request `include_snapshot: true`.
4. Replace local live store with snapshot.
5. Apply subsequent streaming updates.

No replay log is required for v1, but the protocol should support future replay by adding:

```json
{
  "since": "2026-05-05T12:00:00.000Z"
}
```

to `dashboard.subscribe` later.

```python
from fastapi import APIRouter, WebSocket, WebSocketDisconnect
import asyncio

router = APIRouter()


@router.websocket("/ws/workers")
async def worker_websocket(websocket: WebSocket):
    conn = await ws_manager.accept(websocket, role="worker")
    await ws_manager.send(conn.connection_id, make_server_hello(conn.connection_id))

    try:
        auth_msg = await asyncio.wait_for(
            websocket.receive_json(),
            timeout=ws_settings.ws_auth_timeout_seconds,
        )
        await handle_worker_auth(conn.connection_id, auth_msg)

        while True:
            raw = await websocket.receive_json()
            await route_worker_message(conn.connection_id, raw)

    except WebSocketDisconnect:
        await handle_worker_disconnect(conn.connection_id)
    except asyncio.TimeoutError:
        await ws_manager.send(conn.connection_id, make_auth_rejected("AUTH_TIMEOUT", "Authentication timed out"))
        await ws_manager.close(conn.connection_id, code=4002, reason="Authentication timed out")
    except Exception:
        await ws_manager.send(conn.connection_id, make_server_error("INTERNAL_ERROR", "Internal server error", fatal=True))
        await ws_manager.close(conn.connection_id, code=1011, reason="Internal server error")
    finally:
        await handle_worker_disconnect(conn.connection_id)
```

```python
async def route_worker_message(connection_id: str, raw: dict) -> None:
    envelope = WsEnvelope.model_validate(raw)
    conn = await ws_manager.get_connection(connection_id)
    if not conn or not conn.authenticated:
        await ws_manager.send(connection_id, make_server_error("UNAUTHORIZED", "Not authenticated", envelope.message_id, fatal=True))
        await ws_manager.close(connection_id, code=1008, reason="Not authenticated")
        return

    match envelope.type:
        case "client.pong":
            await handle_client_pong(connection_id, envelope)
        case "worker.status":
            await handle_worker_status(connection_id, envelope)
        case "job.claimed":
            await handle_job_claimed(connection_id, envelope)
        case "job.running":
            await handle_job_running(connection_id, envelope)
        case "job.progress":
            await handle_job_progress(connection_id, envelope)
        case "job.log":
            await handle_job_log(connection_id, envelope)
        case "job.complete":
            await handle_job_complete(connection_id, envelope)
        case "job.fail":
            await handle_job_fail(connection_id, envelope)
```

```ts
export function reconnectDelayMs(attempt: number): number {
  const initial = 500;
  const max = 30_000;
  const base = Math.min(max, initial * 2 ** attempt);
  const jitter = 0.8 + Math.random() * 0.4;
  return Math.floor(base * jitter);
}
```

## 5.5 Dashboard WebSocket Client

File:

```txt
frontend/src/lib/ws/dashboardSocket.ts
```

Responsibilities:

- Open `/ws/dashboard`.
- Authenticate.
- Subscribe.
- Handle ping/pong.
- Apply updates into Svelte store.
- Reconnect with backoff.

Implementation skeleton:

```ts
import { browser } from '$app/environment';
import { makeMessage, type WsMessage } from './protocol';
import { reconnectDelayMs } from '$lib/utils/backoff';
import {
  dashboardLiveState,
  applySnapshot,
  applyWorkerStatus,
  applyJobStatus,
  applyJobLog
} from '$lib/stores/dashboardLiveStore.svelte';

let socket: WebSocket | null = null;
let reconnectTimer: ReturnType<typeof setTimeout> | null = null;
let reconnectAttempt = 0;
let manuallyClosed = false;

export interface DashboardSocketOptions {
  topics?: Array<'workers' | 'jobs' | 'logs'>;
  jobIds?: string[];
  workerIds?: string[];
}

export function connectDashboardSocket(options: DashboardSocketOptions = {}) {
  if (!browser) return;

  manuallyClosed = false;
  dashboardLiveState.connection.status = reconnectAttempt === 0 ? 'connecting' : 'reconnecting';
  dashboardLiveState.connection.reconnectAttempt = reconnectAttempt;

  const protocol = location.protocol === 'https:' ? 'wss:' : 'ws:';
  const url = `${protocol}//${location.host}/ws/dashboard`;

  socket = new WebSocket(url);

  socket.onopen = () => {
    dashboardLiveState.connection.status = 'authenticating';
  };

  socket.onmessage = (event) => {
    const message = JSON.parse(event.data) as WsMessage;
    handleMessage(message, options);
  };

  socket.onerror = () => {
    dashboardLiveState.connection.lastError = 'WebSocket error';
  };

  socket.onclose = () => {
    socket = null;

    if (manuallyClosed) {
      dashboardLiveState.connection.status = 'disconnected';
      return;
    }

    scheduleReconnect(options);
  };
}

function handleMessage(message: WsMessage, options: DashboardSocketOptions) {
  switch (message.type) {
    case 'server.hello':
      send(makeMessage('auth.dashboard', { mode: 'session' }));
      return;

    case 'auth.accepted':
      reconnectAttempt = 0;
      dashboardLiveState.connection.status = 'connected';
      dashboardLiveState.connection.connectedAt = new Date().toISOString();
      send(
        makeMessage('dashboard.subscribe', {
          topics: options.topics ?? ['workers', 'jobs', 'logs'],
          job_ids: options.jobIds ?? null,
          worker_ids: options.workerIds ?? null,
          include_snapshot: true
        })
      );
      return;

    case 'server.ping':
      send(
        makeMessage('client.pong', {
          sequence: (message.payload as { sequence: number }).sequence,
          client_time: new Date().toISOString()
        })
      );
      return;

    case 'dashboard.snapshot':
      applySnapshot(message.payload as any);
      return;

    case 'worker.status.updated':
      applyWorkerStatus(message.payload as any);
      return;

    case 'job.status.updated':
      applyJobStatus(message.payload as any);
      return;

    case 'job.log.appended':
      applyJobLog(message.payload as any);
      return;

    case 'server.error':
      dashboardLiveState.connection.lastError = (message.payload as any).message ?? 'Server error';
      return;
  }
}

function send(message: WsMessage) {
  if (socket?.readyState === WebSocket.OPEN) {
    socket.send(JSON.stringify(message));
  }
}

function scheduleReconnect(options: DashboardSocketOptions) {
  dashboardLiveState.connection.status = 'reconnecting';

  const delay = reconnectDelayMs(reconnectAttempt++);
  reconnectTimer = setTimeout(() => {
    connectDashboardSocket(options);
  }, delay);
}

export function disconnectDashboardSocket() {
  manuallyClosed = true;

  if (reconnectTimer) {
    clearTimeout(reconnectTimer);
    reconnectTimer = null;
  }

  socket?.close(1000, 'Manual close');
  socket = null;
}
```

```rust
pub fn reconnect_delay(attempt: u32) -> Duration {
    let initial_ms = 500u64;
    let max_ms = 30_000u64;
    let exp = initial_ms.saturating_mul(2u64.saturating_pow(attempt.min(16)));

}
```

Recommended event channel:

```rust
use tokio::sync::mpsc;

#[derive(Debug, Clone)]
pub enum WorkerWsEvent {
    Status(WorkerStatusPayload),
    JobClaimed(JobClaimedPayload),
    JobRunning(JobRunningPayload),
    JobProgress(JobProgressPayload),
    JobLog(JobLogPayload),
    JobComplete(JobCompletePayload),
    JobFail(JobFailPayload),
}
```

Skeleton:

```rust
use futures_util::{SinkExt, StreamExt};
use tokio::sync::mpsc;
use tokio_tungstenite::connect_async;
use tungstenite::Message;
use serde_json::Value;

pub struct WorkerWsClient {
    pub backend_ws_url: String,
    pub worker_id: String,
    pub token: String,
    pub event_rx: mpsc::Receiver<WorkerWsEvent>,
}

impl WorkerWsClient {
    pub async fn run(mut self) {
        let mut attempt = 0u32;

        loop {
            match self.connect_once().await {
                Ok(_) => {
                    attempt = 0;
                }
                Err(err) => {
                    eprintln!("worker websocket error: {err:?}");
                }
            }

            let delay = crate::ws::backoff::reconnect_delay(attempt);
            attempt += 1;
            tokio::time::sleep(delay).await;
        }
    }

    async fn connect_once(&mut self) -> anyhow::Result<()> {
        let (ws_stream, _) = connect_async(&self.backend_ws_url).await?;
        let (mut write, mut read) = ws_stream.split();

        // Wait for server.hello
        let hello = read.next().await.ok_or_else(|| anyhow::anyhow!("socket closed before hello"))??;
        Self::validate_server_hello(&hello)?;

        // Send auth.worker
        let auth_payload = AuthWorkerPayload {
            worker_id: self.worker_id.clone(),
            token: self.token.clone(),
            client: WorkerClientInfo {
                app_version: env!("CARGO_PKG_VERSION").to_string(),
                platform: std::env::consts::OS.to_string(),
                hostname: hostname::get().ok().and_then(|h| h.into_string().ok()),
            },
        };

        let auth_msg = make_message("auth.worker", auth_payload)?;
        write.send(Message::Text(auth_msg)).await?;

        loop {
            tokio::select! {
                incoming = read.next() => {
                    let Some(incoming) = incoming else {
                        break;
                    };

                    let msg = incoming?;
                    if let Message::Text(text) = msg {
                        self.handle_server_message(&text, &mut write).await?;
                    }
                }

                maybe_event = self.event_rx.recv() => {
                    let Some(event) = maybe_event else {
                        break;
                    };

                    let text = serialize_worker_event(event)?;
                    write.send(Message::Text(text)).await?;
                }
            }
        }

        Ok(())
    }

    async fn handle_server_message<W>(&self, text: &str, write: &mut W) -> anyhow::Result<()>
    where
        W: SinkExt<Message> + Unpin,
        <W as futures_util::Sink<Message>>::Error: std::error::Error + Send + Sync + 'static,
    {
        let value: Value = serde_json::from_str(text)?;
        let message_type = value.get("type").and_then(|v| v.as_str()).unwrap_or("");

        match message_type {
            "server.ping" => {
                let sequence = value["payload"]["sequence"].as_u64().unwrap_or(0);
                let pong = make_message("client.pong", ClientPongPayload {
                    sequence,
                    client_time: chrono::Utc::now(),
                })?;
                write.send(Message::Text(pong)).await?;
            }
            "auth.accepted" => {
                println!("worker websocket authenticated");
            }
            "auth.rejected" => {
                anyhow::bail!("worker websocket auth rejected: {value}");
            }
            "server.error" => {
                eprintln!("server websocket error: {value}");
            }
            _ => {}
        }

        Ok(())
    }

    fn validate_server_hello(msg: &Message) -> anyhow::Result<()> {
        match msg {
            Message::Text(text) => {
                let value: Value = serde_json::from_str(text)?;
                if value.get("type").and_then(|v| v.as_str()) != Some("server.hello") {
                    anyhow::bail!("expected server.hello");
                }
                Ok(())
            }
            _ => anyhow::bail!("expected text server.hello"),
        }
    }
}
```

```rust
    pub async fn job_complete(&self, payload: JobCompletePayload) {
        let _ = self.tx.send(WorkerWsEvent::JobComplete(payload)).await;
    }

    pub async fn job_fail(&self, payload: JobFailPayload) {
        let _ = self.tx.send(WorkerWsEvent::JobFail(payload)).await;
    }
}
```

Integrate with existing flow:

- After REST claim success:
  - Send `job.claimed`
- Before launching coding agent:
  - Send `job.running`
- On phase changes:
  - Send `job.progress`
- On stdout/stderr line:
  - Send `job.log`
- On success:
  - Send `job.complete`
- On failure:
  - Send `job.fail`

Important migration guidance:

- Keep REST completion/failure as source of truth initially.
- WebSocket can duplicate status updates for live UI.
- Once stable, convert WebSocket job terminal messages into authoritative mutations if desired.

Recommended v1 authoritative model:

```txt
REST claim/complete/fail remain authoritative.
WebSocket progress/log/status are live streaming and also persisted when safe.
```

Or, if the backend handlers are implemented with ownership checks and conditional writes, WebSocket can immediately be authoritative for all job state events.

# 7. Error Handling

## 7.1 Message Validation Errors

Server behavior:

- If envelope invalid:
  - Send `server.error` if possible.
  - Close with `1002` or `4003` for repeated/fatal violations.
- If payload invalid:
  - Send `server.error` with `VALIDATION_ERROR`.
  - Keep connection open for non-fatal validation errors.
- If worker identity mismatch:
  - Send `server.error` with `FORBIDDEN`.
  - Close with `4005`.

Example:

```json
{
  "type": "server.error",
  "version": 1,
  "message_id": "srv-err-1",
  "timestamp": "2026-05-05T12:00:00.000Z",
  "payload": {
    "code": "VALIDATION_ERROR",
    "message": "Invalid payload",
    "message_id": "client-msg-1",
    "details": {
      "errors": [
        {
          "field": "payload.job_id",
          "reason": "Field required"
        }
      ]
    },
    "fatal": false
  }
}
```

---

## 7.2 Authentication Timeout

If no valid auth message is received within `WS_AUTH_TIMEOUT_SECONDS`:

1. Send `auth.rejected` with `AUTH_TIMEOUT`.
2. Close with code `4002`.

---

## 7.3 Heartbeat Timeout

If no `client.pong` within timeout:

1. Close socket with code `1001` or custom `4000`.
2. For worker:
   - Mark worker offline only if the disconnected connection is still the active worker connection.
   - Broadcast `worker.status.updated`.
3. Dashboard will reconnect automatically.

---

## 7.4 Worker Disconnect During Running Job

Do not immediately fail job.

Server should:

1. Mark worker `offline`.
2. Keep job status as `running`.
3. Existing scheduler/timeout process decides whether to mark job `timed_out`.
4. If same worker reconnects and resumes, it sends `worker.status` and job progress.

Optional future improvement:

- Add worker resume message:

```txt
worker.resume
```

Payload:

```json
{
  "worker_id": "worker_123",
  "running_job_ids": ["job_456"]
}
```

Not required for v1.

---

## 7.5 Duplicate Messages

Workers may resend events after reconnect.

Backend should treat these idempotently where possible:

- `job.progress`: overwrite latest progress.
- `job.log`: use `job_id + sequence` as idempotency key.
- `job.complete`: terminal transition should be idempotent if same worker/result.
- `job.fail`: terminal transition should be idempotent if same worker/error.

Add optional DynamoDB item for message de-duplication later:

```txt
PK = WSMSG#worker_123
SK = MSG#message_id
```

Not required for v1 unless duplicate terminal events are likely.

---

## 7.6 Message Size Limits

Server should reject overly large messages.

Config:

```txt
ws_max_message_bytes = 65536
job.log.line maxLength = 16000
progress.message maxLength = 2000
```

FastAPI/Starlette does not expose all frame size controls directly, so enforce after receiving JSON:

```python
raw_text = await websocket.receive_text()
if len(raw_text.encode("utf-8")) > ws_settings.ws_max_message_bytes:
    await ws_manager.close(connection_id, code=1009, reason="Message too large")
```

Prefer `receive_text()` + `json.loads()` over `receive_json()` if strict size enforcement is required.

---

## 7.7 Rate Limiting

Initial practical limits:

```txt
worker.status: max 1 per 5 seconds
job.progress: max 1 per 1 second per job
job.log: max 20 per second per job
dashboard.subscribe: max 1 per 5 seconds
```

Implement lightweight in-memory token bucket in:

```txt
backend/app/services/ws_rate_limiter.py
```

For v1, at minimum throttle log broadcast to avoid overwhelming dashboards.
